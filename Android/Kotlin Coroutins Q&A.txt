1)What is coroutine?

ans: Coroutines are lightweight threads you can use in android to simplify code 
	 that executes asynchronously. coroutine helps to manage long-running task.
	
	advantages of coroutine
	
	1) Executed with in a thread 
	
	we can use many coroutine inside single thread 
	
	2)Coroutines are suspendable

	was started in one thread can easily switch other the thread.

	a few KB of memory.
	
	While Using Rx, it requires a lot of effort to get it enough, to use it

3)How many scope in kotlin coroutine?

ans: There are basically 3 scopes in kotlin coroutine

	1)Global Scope 2) LifeCycle Scope 3) ViewModel Scope
	
	1) Global Scope: When Coroutines are launched within the global scope, 
					 they live long as the application does.Hence it will
					 leads to memory leak issue.
					 
	2)LifeCycle Scope: Lifecycle scope is the same as the global scope,
					 but the only difference is that when we use the lifecycle 
					 all the coroutines launched within the activity 
					 also dies when the activity dies.

	3)ViewModel Scope: It is also the same as the lifecycle scope, only 
					   difference is that the coroutine in this scope will live 
					   as long the view model is alive. ViewModel is a class that
					   manages and stores  the UI-related data by following the 
					   principles of the lifecyclesystem in android. 
				


4)What are Dispatchers in android?

ans: Dispatchers help coroutines to decide on which  thread  work has
	 to be done. we can passed as arguments to scope and there are 
	 mainly 4 types of dispatchers.
	 
	 1)Main Dispatcher    2)IO Dispatcher 
	 3)Default Dispatcher 4)Unconfined Dispatcher
	 
	 1) Main Dispatcher:
	 
	 It starts the coroutine in the main thread. It is mostly used when we 
	 need to perform the UI operations within the coroutine, as UI can only
	 be changed from the main thread(also called the UI thread).
	 
	 2) IO Dispatcher:
	 
	 Its start coroutine in the IO thread, it is used to perform all the data 
	 operations such as networking, reading, or writing from the database.
	 
	 3) Default Dispatcher:
	 
	 Its start coroutine in the Default thread,If we are planning to do complex 
	 and long- running application, which can block the main thread and freez 
	 the Ui.
	 
	 eg you need to sort list of 10,000	elements so do that in default dispatcher
	 so it will not block you ui thread.
	 
	 4) Unconfined Dispatcher:
	 
	 As the name suggests unconfined dispatcher is not confined to any specific
	 thread.

15) What is an inline function in Kotlin?

ans: Inline function instruct compiler to insert complete body of the function
	 wherever that function got used in the code.To use an Inline function, all
	 you need to do is just add an inline keyword at the beginning of the 
	 function declaration.	 
	 Advantage of inline function: Function call overhead doesn't occur. 
	 Less overhead and faster program execution.

	So, when to make the function inline and when not:

	When the function code is very small, it's a good idea to make the function
	inline.
	When the function code is large and called from so many places, it's a bad 
	idea to make the function inline, as the large code will be repeated again 
	and again.

26)Explain the use case of let,run,also,apply

ans:Explain the use-case of let, run, with, also, apply in Kotlin.
	"There are five scoped functions in Kotlin: let, run, with, also and apply. 
	Let’s go through them one by one.

	But before going through these examples, let’s consider a Model class “Person”"
	
	class Person() {
		var name: String = "Abcd"
		var contactNumber: String = "1234567890"
		var address: String = "xyz"
	   fun displayInfo()  = print("\n Name: $name\n " +
				"Contact Number: $contactNumber\n " +
				"Address: $address")
	}	
	  
	
	Scoped Functions in Kotlin
	By definition, Scoped functions are functions that execute a block of code within
	the context of an object.  

	
	 1) let: context object: it
		   return value: lambda result
		   
		The let function is frequently used for null safety calls. for null safety
		we use safe call operator(?.) with let. It only runs the block with a non
		null value.
		
		example 
		list.size?.let{
			
		}
	
     2)apply: context object: this
			return value: context object
		
		Apply with the help of this we can set values to member object. 	
		we don't need to do this obj.Fname = "Fname" and obj.Lname = "Lname.

	  	// Normal approach
		fun createIntent(intentData: String, intentAction: String): Intent {
			val intent = Intent()
			intent.action = intentAction
			intent.data = Uri.parse(intentData)
			return intent
		}

	// Improved approach, by using apply
		fun createIntent(intentData: String, intentAction: String) =
			Intent().apply {
				action = intentAction
				data = Uri.parse(intentData)
			}	
		
	 3)run: context object:this
		The “run” operator is similar to the “let” operator in terms of accepting
		a return value that is different from the object on which the scope function 
		is being applied to. Hence, a “run” operator can be used to initialize an 
		object and return the result of it.
		
		private fun performRunOperation() {
			Person().run {
				name = "Asdf"
				contactNumber = "0987654321"
				return@run "The details of the Person is: ${displayInfo()}"
			}
		}
		
		output:
		 Name: Asdf
		 Contact Number: 0987654321
		 Address: xyz

		Note : Person Object Refer from line 101
		
		apply vs run
		
		The “also” function is similar to the let functionality only in terms of 
		referring to the context of the object as “it” and not “this” and also in
		providing null safety checks:
		
		private fun performAlsoOperation() {
			val name = Person().also { currentPerson ->
			print("Current name is: ${currentPerson.name}\n")
			currentPerson.name = "modifiedName"
			}.run {
				"Modified name is: $name\n"
			}
			print(name)
		}
		
		output:
		Current name is: Abcd
		Modified name is: modifiedName
		
	 4)with: The “with” operator is completely similar to the run operator that we 
			 just discussed. It also refers to the context of the object as “this”,
			 similar to how the “run” operator uses it.	
			 
		with vs run:	 
			  
		So, if “with” is the same as “run”, what’s the difference? How should we 
		choose between these two? 
		
		Now, this is an interesting case. Let’s consider a case where a Person object
		can be nullable. 	  
		
		we can see that the context of the object referred to as “this” is a nullable 
		type of Person. And hence, to correct this, we need to change the code as:	
		
		private fun performWithOperation() {
			val person: Person? = null
			with(person) {
				this?.name = "asdf"
				this?.contactNumber = "1234"
				this?.address = "wasd"
				this?.displayInfo()
			}
		}
		
		“also” vs “let”
		
		We can see that let accepts a return statement whereas “also” does not accept 
		a return statement and always returns the same object which it is referring to.

29)What is difference between Cold flow and Hot flow?
ans: 
	1) Cold Flow:
	   A cold flow starts producing values only when collected.
	   Each collector gets its own independent stream of values.	
	example: Netflix on-demand — every time you hit play, the show starts from
			the beginning.  	
	   	
	fun coldFlow(): Flow<Int> = flow {
    for (i in 1..3) {
        delay(500)
        emit(i)
        println("Emitting $i")
      }
    }

	fun main() = runBlocking {
		println("Collector 1 starts")
		coldFlow().collect { println("Collector 1 received $it") }

		println("Collector 2 starts")
		coldFlow().collect { println("Collector 2 received $it") }
	}
	
	Output:
	
	Collector 1 starts
	Emitting 1
	Collector 1 received 1
	Emitting 2
	Collector 1 received 2
	Emitting 3
	Collector 1 received 3

	Collector 2 starts
	Emitting 1
	Collector 2 received 1
	Emitting 2
	Collector 2 received 2
	Emitting 3
	Collector 2 received 3

   2) Hot flow
	  A hot flow emits values regardless of collectors.	
	 Collectors only receive values from the moment they start collecting.
     Example: StateFlow, SharedFlow.
	 👉 Think of it like live TV — if you join late, you miss what was already 
		broadcast.
		
	fun main() = runBlocking {
		val stateFlow = MutableStateFlow(0)

		// Collector 1
		launch {
			stateFlow.collect { println("Collector 1 received $it") }
		}

		delay(1000)
		stateFlow.value = 1
		stateFlow.value = 2

		// Collector 2 joins late
		launch {
			stateFlow.collect { println("Collector 2 received $it") }
		}

		delay(1000)
		stateFlow.value = 3
	}
	output:
	
	Collector 1 received 0
	Collector 1 received 1
	Collector 1 received 2
	Collector 2 received 2   <-- joined late, gets latest value immediately
	Collector 1 received 3
	Collector 2 received 3
