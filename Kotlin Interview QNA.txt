2)How to create singleton in Kotlin?

ans: Just use object 
	e.g object SomeSingleton("text")
	
	The above Kotlin object will be compiled to the following equivalent  code:
	
	class SingletonDemo(val text: String) private constructor(){
	
	companion object(){
	
		@Voilatile
		private lateinit var instance: SingletonDemo
		
		fun getInstance(text: String):SingletonDemo{
			
			Synchronized(this){
			   if(!::instance){
				  instance = SingletonDemo(text)	
			   }	
			   return instance		
			}
		}
		
		fun show(){
			println("This is singleton class")
		}
	}

}

fun main(){
	SingletonDemo.getInstance("text").show()
}

29)How ViewModel Internally works

ans: When an Activity/Fragment is destroyed and recreated (rotation, config change),
	all instance variables are lost. ViewModel survives this by being stored outside
	the Activity/Fragment lifecycle.
	 
	ViewModel works in 3 steps:
	1)Storage — Every Activity holds a ViewModelStore, which is just a HashMap of
   	  ViewModels.
	
	2)Survival — When screen rotates, Android calls onRetainNonConfigurationInstance()
	  before destroying the Activity, which saves the ViewModelStore. The new Activity
	  gets the same store back.
	  
	3)Retrieval — ViewModelProvider checks the store first. If ViewModel exists → returns
 	  same instance. If not → creates new one via Factory."
	
	
	If they ask "when does it get destroyed?"

	"Only when the Activity truly finishes — like the user pressing Back — not on rotation.
 	That's when onCleared() is called."
	
	If they ask about coroutines

	"viewModelScope is a CoroutineScope with SupervisorJob + Dispatchers.Main. It's 
	automatically cancelled inside onCleared(), so no manual cleanup needed."
	
30)	How Glide Works Internally?

ans: 1)Request — When you call Glide.with(context).load(url).into(imageView), Glide builds
	   a Request object and attaches it to the ImageView.
	
	 2)Cache Check — Before hitting the network, Glide checks two cache layers. First Memory
	   Cache (active/LRU), if not found then Disk Cache. If found in either → loads instantly.
	   
	 3)Network/Decode — If not cached, Glide fetches from network via HttpUrlConnection or 
	   OkHttp, then decodes and downsamples the bitmap to exactly the ImageView size — not 
	   the original image size.
	   
	 4)Display & Cache — Decoded bitmap is shown in the ImageView, then stored in both Disk and
   	   Memory cache for future use."  
	   
	If they ask about lifecycle

	"Glide attaches a hidden Fragment to the Activity/Fragment to observe lifecycle. When Activity
	 is destroyed, Glide automatically cancels pending requests — no memory leaks."   
	    
	If they ask about cache layers

	"Glide has 4 cache layers in order — Active Resources (currently displayed), Memory Cache (LRU),
 	 Resource Disk Cache (transformed), Data Disk Cache (original raw data)."	   
	   
	If they ask why Glide is memory efficient

	"Because Glide downsamples the image to match the exact ImageView dimensions before loading into
 	memory. A 4K image shown in a 100x100 view is decoded as 100x100 — not full size."   
	   
31)How LiveData Works Internally?

ans: LiveData works in 3 steps:
	   
	1)Wrapping — Every observer is wrapped in a LifecycleBoundObserver which listens to the lifecycle.
 	It marks itself active only when state is STARTED or RESUMED.   
	
	2)Version System — LiveData maintains a version counter. Each observer tracks the last version it
  	saw. When observer becomes active again, it compares versions — if it missed an update, it gets the
	latest value immediately.
	   
	3)Auto Cleanup — When lifecycle hits DESTROYED, the observer is automatically removed from the map
	— no memory leaks."   
	   
	 If they ask setValue vs postValue

	"setValue must run on main thread, directly updates value and notifies observers. postValue can be
	called from background thread — it internally posts a runnable to main thread and calls setValue."  
	
	If they ask observe vs observeForever

	"observe ties to a lifecycle owner — auto removes on DESTROYED. observeForever has no lifecycle 
	awareness — you must manually call removeObserver or it leaks."
	
	If they ask why observer gets last value on rotation

	"Because of the version system. When new Activity subscribes, observer's lastVersion is -1 but
	LiveData's version is already 1 — so it immediately delivers the current value."
